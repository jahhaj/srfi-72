A portable implementation of R6RS libraries and macros for Scheme.

=20 This implementation differs from the Ghuloum-Dybvig (psyntax)=20 implementation in various aspects, some of which are described here.

R6RS COMPLIANCE NOTES:
=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D

SYNTAX-CASE:=20
------------  =20

 FULLY R6RS-COMPLIANT, WITH THE FOLLOWING OPTIONAL OR RECOMMENDED FEATURES:
 =20
   * Expansion algorithm (r6rs optional):
  =20
       We use a renaming algorithm instead of the mark-antimark
       algorithm described in the draft.  The difference will not=20
       matter for r6rs-conforming macros.

   * Wrapped =3D Unwrapped (r6rs optional):
  =20
       A wrapped syntax object is the same as an unwrapped syntax=20
       object, and can be directly manipulated using car, cdr, ...=20
       without syntax-case deconstruction. =20
       This should make porting of legacy low-level macros,
       e.g., explicit renaming or define-macro, easier.
       Macros depending on this feature wil not be=20
       portable to all r6rs implementations. =20
                =20
   * Lexical scoping violation detection (r6rs recommended):
  =20
       We treat a violation of the following restriction in chapter 8
       as a syntax violation:
      =20
       A definition in the sequence of forms must not define any=20
       identifier whose binding is used to determine the meaning of the
       undeferred portions of the definition or any definition that=20
       precedes it in the sequence of forms.      =20

LIBRARIES:
----------

 FULLY R6RS-COMPLIANT WITH ONE EXCEPTION (SEE BELOW) WITH THE FOLLOWING=20
 OPTIONAL FEATURES:=20
  =20
   * Phase semantics (r6rs optional):
  =20
       - A syntax violation is raised at expand-time if a binding is
         referenced outside its declared levels.  We do this for=20
         both variable references and macro references.  As we are=20
         entitled to do by R5.97RS, we also check levels for some
         references that occur when free-identifier=3D? is called.
         Specifically:=20
         =20
         A syntax error is raised if free-identifier=3D? succeeds=20
         but either argument is outside its declared level.  This is suffic=
ient=20
         to ensure that literals such as ... in syntax-case are used at the=
=20
         correct level.  See the examples file for more discussion, and for=
=20
         an explanation of why the levels should not be checked if the=20
         comparison fails. =20
        =20
         Note: This choice raises a syntax violation when MzScheme and Chez
               would disagree, thus ensuring that a program blessed by this
               expander will be maximally portable. =20
               =20
       - We start each expansion of a library form by removing visits of li=
braries=20
         in any phase and instances of libraries in phases above 0.=20
         We start each execution of a toplevel program by removing instance=
s of
         libraries in phase 0.
               =20
       - While expanding a library form, we use an instance/visit at any ph=
ase as=20
         an instance/visit at any other phase.    =20
                  =20
       - We may create instances/visits of a library at more phases than=20
         strictly required to satisfy references, but only in the case - of=
=20
         which the semantics is unspecified in r6rs - where a library is im=
ported=20
         for some phase but none of its bindings is used in that phase. =20
         Specifically:
               =20
         During expansion of a library or program, each library imported=20
         at phase n > 0 is instantiated once, and each library imported=
=20
         at phase n >=3D 0 is visited once.  =20
               =20
         During execution of a program, each library imported=20
         at phase n =3D 0 is instantiated once.
      =20
 KNOWN R6RS-NONCOMPLIANCE:      =20
      =20
   * Version name and reference are syntax-checked, but otherwise ignored.=
=20
=20
ADDITIONAL FEATURES:
=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=3D=20

   * Separate and joint code organization and expansion:
  =20
       Libraries and programs may be in separate files or in the same file,
       and may be separately or jointly expanded.=20
       See example scripts in examples.scm.

   * Repl:
  =20
       A REPL is provided that is integrated with the r6rs library system. =
=20
       Libraries and programs can be defined at, imported into and run in t=
he REPL toplevel.=20
       The ex:repl procedure expands and evaluates a sequence of forms one =
by one.=20
       See example REPL session examples.scm.

   * Dynamic loading of libraries:

       A non-standard library (rnrs load) is provided containing LOAD, whic=
h can be
       used for dynamic loading of any code, including dynamic loading=20
       of R6RS libraries.
       See example in examples.scm.
      =20
   * Reflection:
   =20
       A library (rnrs eval reflection (6)) is provided. =20
       It exports the procedure (environment-bindings ).
       Displays for each binding whether it is a variable or macro, its
       source library, and its levels.
       Very useful for library development and debugging.=20
       See examples of use in examples.scm. =20
      =20
   * Explicit renaming:
    =20
       A compatible explicit renaming library (explicit-renaming) has=20
       been included.  Although such a library is definable in syntax-case
       implementations that follow the algorithm described in the r6rs draf=
t,=20
       it would be inherently inefficient, suffering from an unavoidable
       increase in complexity class over comparable syntax-case macros. =20
       Our algorithm does not have this problem, is in fact nothing but
       explicit renaming behind the scenes, and allows us to expose
       explicit renaming efficiently.
       See further documentation in standard-libraries.scm and=20
       tutorial in examples.scm. =20
